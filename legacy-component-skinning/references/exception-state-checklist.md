# 异常态 / 降级态 / 罕见反馈检查清单

> 很多风格问题只会在非 happy path 才出现，如果不主动复现，这些界面会长期沿用旧皮肤。

## 覆盖目标

把以下状态视为必须检查的 surface，而不是“顺手看一下”：

- 后端不可用
- 请求失败 / 超时
- 权限不足 / 功能未开启
- 表单校验失败
- 空数据与错误提示同时存在
- 只读锁定 / 不可编辑
- 系统级 warning / error banner

## 安全边界

- 只使用本地、隔离测试环境、浏览器级 request blocking / mock、URL 参数、测试账号、feature flag 等可逆低风险手段复现异常态。
- 禁止为了看异常 UI 去停共享服务、改公共配置、污染公共数据、破坏真实后端或影响其他人正在使用的环境。
- 如果某个异常态只能通过高风险手段复现，停止自动化操作，在最终汇报中标明未验证原因，并请求人工批准或替代方案。

## 入口发现法

优先从以下路径枚举异常态：

- 浏览器 request blocking、接口 mock、代理层固定错误返回
- 环境开关、URL 参数、测试账号、mock 数据
- 点击提交后表单校验失败
- 进入没有权限的页面或点击无权限动作
- 隔离测试环境中的服务 stub、错误地址注入或配置缺失模拟
- 列表无数据且伴随错误提示

## 每个异常态都要记录

- 触发条件
- 是否可稳定复现
- 出现位置：page / card / modal / form / inline
- 反馈类型：banner / alert / toast / inline error / empty-error hybrid
- 是否有图标、操作按钮、帮助链接、关闭按钮

## 必查问题

- 异常 banner 的图标、标题、正文、边框、背景、圆角和间距是否协调
- 长错误文案是否换行正常，不会挤坏布局
- 带按钮或链接的错误提示是否仍保持对齐
- 异常态出现后，页面整体层级是否仍成立，不会把主要表单或卡片挤坏
- 表单校验失败后的 inline error、输入框边框、help text 是否在同一节奏
- 系统降级时，提示条和主登录框 / 主卡片之间的间距是否合理

## 最小证据

每个关键异常态至少保留：

- 1 张正常态截图
- 1 张异常态截图
- 1 条触发条件说明

如果无法稳定复现，也要在最终汇报中明确写出“哪些异常态未能验证、为什么未能验证”。

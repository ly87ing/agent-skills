# 复合业务场景检查清单

> 仅把 Button / Input / Table 等原子组件调像还不够。真实项目里的大量问题发生在“组件被拼起来之后”。

## 为什么要有这份清单

以下问题通常不会在组件 playground 中暴露，却会在真实业务页里集中出现：

- 空值下拉触发器宽度被收窄
- placeholder-only 状态下，选中内容或展开菜单难以识别
- 混合表单里的 label、input、textarea、switch、select 基线不齐
- 弹窗中的说明块、表单块、footer 按钮组不在同一布局节奏
- 帮助文案、提示块、多行说明在视觉上脱队
- 卡片内横向选项组在长文案或窄容器下发生截断、挤压或巨大空白
- 点击后才出现的弹窗、tab panel、折叠区在默认页检查中被漏掉
- 特殊错误横幅、降级提示条、罕见反馈卡片只在异常条件下出现，因此被漏检

## 必查复合场景

至少覆盖以下 7 类中的 4 类；如果页面实际存在，就必须检查：

### 1. 列表页工具栏 / 筛选区
- 搜索框 + 下拉 + 日期范围 + 操作按钮的混排
- 状态筛选下拉在无值、默认值、已选值下的宽度
- 表头上方统计文案与按钮组的对齐

### 2. 配置表单 / 设置页
- 左侧 label + 右侧控件的列宽与基线
- 输入框、密码框、select、textarea、switch 的纵向节奏
- 必填星号、说明文案、错误文案、placeholder 的对齐

### 3. 弹窗 / 抽屉表单
- header 与关闭按钮
- 内部说明卡片 / 提示块 / banner
- 表单块的左右内边距、label 宽度、按钮 footer
- 不同字段宽度是否有统一规则

### 4. 上传 / 接入 / 下载链接类弹框
- 多个同构输入框是否左对齐、等宽、等间距
- 说明文案是否与输入组保持一致缩进
- 空值时输入框是否被异常压缩

### 5. 日志 / 审计 / 监控检索页
- 时间范围、筛选条件、导出按钮的横向对齐
- 顶部描述文字、筛选 chips、表格之间的留白
- 表格标题、分页、状态筛选之间的节奏

### 6. 只读 / 禁用 / 灰度配置页
- disabled / read-only 控件的宽度、颜色层级和占位文本可读性
- 开关、选择器、版本号输入框在禁用态下是否仍保持布局稳定

### 7. 卡片内横向选项组 / segmented rail / 特性分组
- 文案 + 徽标/标签 + 选中态 chip 的横向组合
- 长文案、多个分组、窄容器下是否发生裁切或大面积留白
- 组内是否允许 wrap、scroll，或是否有明确的 min-width 策略
- 选中态徽标、按钮、标签与主文案是否保持基线和节奏一致

## 每个复合场景都要检查的点

- 触发器宽度：有值、无值、placeholder-only 时宽度是否一致
- 下拉面板：展开后选项宽度、选中态、hover 态、滚动区域是否清晰
- label 对齐：单行 label、多行 label、带必填星号 label 是否对齐
- control 列宽：同组 input / select / textarea 是否有统一视觉宽度
- 说明文案：help text、提示块、错误文案的缩进和换行是否稳定
- 内容节奏：信息 banner、表单卡片、footer 按钮之间的留白是否统一
- 按钮组：主次按钮的高度、间距、基线、右对齐规则是否统一
- 横向排列策略：是否错误使用 `space-between`、`nowrap`、固定宽度导致截断或空洞
- 交互入口覆盖：是否真的点开了更多/高级/编辑/修改密码后再检查

## 极端状态

不要只看“有正常值”的 happy path，至少覆盖：

- 空值
- placeholder-only
- 长 placeholder
- 长选项文本
- 长分组标题 / 长特性名
- disabled
- read-only
- 多行帮助文案
- 仅有提示块、尚未填写任何值的初始态
- 小宽度容器或父容器可滚动场景

## 最小验收证据

每类复合场景至少保留：

- 1 张默认态截图
- 1 张空态或 placeholder 截图
- 1 张展开态 / 编辑态 / 弹窗态截图

最终汇报里应明确写出已检查的复合场景，而不是只说“表单、弹窗已检查”。
